Dependency Inversion principle(DIP):
      Dependency Inversion Principle (DIP)
      Definition
            The Dependency Inversion Principle (DIP) is the "D" in the SOLID principles.
            It states:
                  High-level modules should not depend on low-level modules. Both should depend on abstractions.
                  Abstractions should not depend on details. Details should depend on abstractions.

      Key Terms
            High-Level Module
                  Contains business logic and application rules.
            Example:
                  public class UserService {
                      // Business logic
                  }

            Low-Level Module
                  Performs specific implementation details.
                  Example:
                        public class EmailNotificationService {
                            // Sends email
                        }

            Abstraction
                  Usually an interface that defines a contract.
                  Example:
                        public interface NotificationService {
                            void send(String message);
                        }

            Problem Without DIP
                  public class UserService {
                        private EmailNotificationService emailService = new EmailNotificationService();
                        public void registerUser() {
                                emailService.send("Welcome");
                        }
                  }
            Issues
                  Tight coupling between classes.
                  Difficult to replace email with SMS or Push notifications.
                  Harder to test.
                  Changes in low-level modules may affect high-level modules.

            Solution Using DIP
                  public interface NotificationService {
                      void send(String message);
                  }

                  public class EmailNotificationService
                          implements NotificationService {
                  
                      @Override
                      public void send(String message) {
                          System.out.println("Email sent");
                      }
                  }

                  public class UserService {
                  
                      private final NotificationService notificationService;
                  
                      public UserService(NotificationService notificationService) {
                          this.notificationService = notificationService;
                      }
                  
                      public void registerUser() {
                          notificationService.send("Welcome");
                      }
                  }

      Where Is the Dependency Inverted?
            Before DIP
                        UserService
                              ↓
                        EmailNotificationService
                  High-level module depends directly on low-level module.
            After DIP
                        UserService
                              ↓
                        NotificationService
                              ↑
                        EmailNotificationService
                  Both classes depend on the abstraction.
            The dependency direction is reversed (inverted), which is why it is called Dependency Inversion Principle.

      Benefits of DIP
            1. Loose Coupling
                  High-level modules are not tied to specific implementations.
            2. Easy to Extend
                  New implementations can be added without modifying existing business logic.
                  Example:
                        public class SmsNotificationService
                                implements NotificationService {
                        }
                  UserService remains unchanged.
            3. Better Testability
                  Mock implementations can be injected during testing.
                  public class MockNotificationService
                          implements NotificationService {
                  }
            4. Improved Maintainability
                  Changes in implementation details do not affect business logic.



Dependency Injection (DI)
      Before understanding Dependency Injection, you must first understand Dependency.
    What is a Dependency?
      Suppose you have a service that sends emails.
          public class EmailService {
              public void sendEmail(String message) {
                  System.out.println("Sending Email: " + message);
              }
          }
      Now a user service wants to use it.
          public class UserService {
              private EmailService emailService = new EmailService();
              public void registerUser() {
                  emailService.sendEmail("Welcome!");
              }
          }
      Here:
          EmailService
      is a dependency of UserService
      because UserService depends on it to perform its work.

What's the Problem?
    Notice this line:
        private EmailService emailService = new EmailService();
    UserService is creating its own dependency.
    This causes tight coupling.
        UserService ----> EmailService
    If tomorrow you want:
        EmailService
        SmsService
        WhatsAppService
    you must modify UserService.
    This violates OCP and makes testing harder.

Dependency Injection Idea
    Instead of creating dependencies yourself:
      private EmailService emailService = new EmailService();
    someone else provides them.
    This is called Dependency Injection.
        public class UserService {
            private EmailService emailService;
            public UserService(EmailService emailService) {
                this.emailService = emailService;
            }
        }
    Now UserService doesn't create the dependency.
    It only uses it.

Real World Analogy
  Imagine you're a driver.
  Without DI:
      You buy the car
      You maintain the car
      You create the car
  With DI:
      Someone gives you a car
      You simply drive it
  You focus on your responsibility.


In Spring Boot
    Spring's IoC Container creates objects and injects them where needed.
    Step 1:
      Email Service
          @Service
          public class EmailService {
          
              public void sendEmail(String message) {
                  System.out.println(message);
              }
          }

    Step 2:
      Inject into UserService
        @Service
        public class UserService {
            private final EmailService emailService;
            public UserService(EmailService emailService) {
                this.emailService = emailService;
            }
            public void registerUser() {
                emailService.sendEmail("Welcome");
            }
        }
      
      What happens internally?
      Spring sees:
          @Service
          public class EmailService
      and creates an object:
          EmailService emailService = new EmailService();
      inside its container.
      Then it sees:
          public UserService(EmailService emailService)
          and injects that object automatically.
      Conceptually:
          EmailService emailService = new EmailService();
          UserService userService = new UserService(emailService);
      Spring does this for you.
      Why is it Called Injection?
          Because the dependency is literally injected into the class.
            new UserService(emailService);
          The dependency enters from outside.

Types of Dependency Injection
    There are three common types.
    1. Constructor Injection (Recommended)
    @Service
    public class UserService {
        private final EmailService emailService;
        public UserService(
        EmailService emailService) {
            this.emailService = emailService;
        }
    }
    Spring injects through the constructor.
    Advantages
        Immutable dependency (final)
        Easy testing
        Mandatory dependency
        Recommended by Spring
    This is what you should use most of the time.

    2. Setter Injection
        @Service
        public class UserService {
        
            private EmailService emailService;
        
            @Autowired
            public void setEmailService(
                    EmailService emailService) {
        
                this.emailService = emailService;
            }
        }
      Spring injects using a setter method.
      Useful for optional dependencies.

3. Field Injection
      @Service
      public class UserService {
          @Autowired
          private EmailService emailService;
      }
      Spring injects directly into the field.
      Works, but generally discouraged.
      Problems
          Harder to test
          Hidden dependencies
          Cannot make field final

Why Constructor Injection is Preferred
    Compare:
        Field Injection:
            @Service
            public class UserService {
                @Autowired
                private EmailService emailService;
            }
        You cannot immediately see what the class requires.

        Constructor Injection
            @Service
            public class UserService {
                private final EmailService emailService;
                public UserService(
                        EmailService emailService) {
                          this.emailService = emailService;
                }
            }
      The dependency is explicit.
      Any developer can instantly see:
      UserService needs EmailService

How DI Helps with Testing
  Without DI:
      public class UserService {
          private EmailService emailService =
                  new EmailService();
      }
  Testing is difficult because you cannot replace the dependency.
  
  With DI:
    public class UserService {
        private final EmailService emailService;
        public UserService(
                EmailService emailService) {
            this.emailService = emailService;
        }
    }
  
  You can pass a mock:
    EmailService mockEmailService =
            Mockito.mock(EmailService.class);
    UserService userService =
            new UserService(mockEmailService);
    Very easy to test.

Dependency Injection vs Inversion of Control (IoC)
    These are related but not identical.
    IoC
    A principle:
        The framework controls object creation.
    DI
    A technique:
        The framework provides dependencies to objects.
    Think:
      IoC = Big concept
      DI = One way to achieve IoC
    In Spring:
      Spring Container
              ↓
      Creates Beans
              ↓
      Injects Dependencies

Interview Definition
    Dependency Injection is a design pattern in which a class receives its dependencies from an external
    source instead of creating them itself. In Spring Boot, the IoC container creates beans and injects them
    into dependent classes, reducing coupling and improving testability, maintainability, and flexibility.

Memory Trick
    Dependency  -> Something a class needs
    Injection   -> Provided from outside

Dependency Injection
    = Give a class what it needs,
      instead of letting it create it.

This is one of the most important concepts in Spring Boot because almost everything (@Service, @Repository, @Controller, @Component, @Autowired) revolves around Dependency Injection and the Spring IoC container.

| Dependency Inversion Principle | Dependency Injection               |
| ------------------------------ | ---------------------------------- |
| Design Principle               | Implementation Technique           |
| Tells what design to follow    | Tells how to provide dependencies  |
| Part of SOLID                  | Design Pattern / Framework Feature |
| Depends on abstractions        | Injects implementations            |


Question: The dependency Injection is indeed called as the dependency Inversion principle?
Answer:
      Not exactly.
      They are related, but they are not the same thing.
      Dependency Inversion Principle (DIP)
            This is the "D" in SOLID.
            It states:
                  High-level modules should not depend on low-level modules. Both should depend on abstractions.
            and
                  Abstractions should not depend on details. Details should depend on abstractions.
      
      Relationship
            Think of it like this:
                  | Concept                        | What it is                                                           |
                  | ------------------------------ | -------------------------------------------------------------------- |
                  | Dependency Inversion Principle | A design principle (SOLID)                                           |
                  | Dependency Injection           | A technique/pattern used to achieve that principle                   |
                  | Spring IoC Container           | A framework feature that performs dependency injection automatically |
                  
So when Spring does:
            @Service
            public class UserService {
                private final NotificationService notificationService;
                public UserService(NotificationService notificationService) {
                    this.notificationService = notificationService;
                }
            }
      Spring is performing Dependency Injection, which helps your code follow the Dependency Inversion Principle.
      A simple way to remember:
            DIP tells you WHAT kind of design you should have.
            DI is HOW you commonly implement that design.
