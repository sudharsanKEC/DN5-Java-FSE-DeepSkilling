What is SOLID principles?
  The SOLID principles are a set of five design guidelines in object-oriented programming intended to make software designs more understandable, flexible, and maintainable. They were popularized by Robert C. Martin ("Uncle Bob") and are foundational for writing high-quality, scalable code.

1. S – Single Responsibility Principle (SRP)
    "A class should have one, and only one, reason to change."
    This means a class should focus on a single task or functionality. If a class handles multiple unrelated responsibilities, it becomes tightly coupled and difficult to maintain. When one responsibility changes, you risk breaking the others.

2. O – Open/Closed Principle (OCP)
    "Software entities (classes, modules, functions) should be open for extension but closed for modification."
    You should be able to add new functionality to a system without changing the existing, tested code. This is typically achieved by using interfaces or abstract classes, allowing you to "plug in" new behavior rather than altering stable logic.

3. L – Liskov Substitution Principle (LSP)
    "Objects of a superclass should be replaceable with objects of its subclasses without breaking the application."
    If class B is a subclass of A, you should be able to use B anywhere you use A without the program failing or behaving incorrectly. Subclasses must honor the contract established by the parent class.

4. I – Interface Segregation Principle (ISP)
    "Clients should not be forced to depend upon interfaces that they do not use."
    Instead of one "fat" interface with many methods, it is better to create multiple, smaller, specific interfaces. This prevents classes from being forced to implement methods they don't need, which keeps the code clean and decouples components.

5. D – Dependency Inversion Principle (DIP)
    "Depend upon abstractions, not concretions."
    High-level modules (the core business logic) should not depend on low-level modules (like database drivers or network APIs). Both should depend on abstractions (interfaces). This allows you to swap out implementations (e.g., changing from a SQL database to a NoSQL database) without impacting your business logic.
