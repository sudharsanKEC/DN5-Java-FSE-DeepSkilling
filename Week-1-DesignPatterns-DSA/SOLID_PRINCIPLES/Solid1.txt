The 'S' in SOLID stands for the Single Responsibility Principle(SRP).

SRP:
    The Single Responsibility Principle (SRP) states that a class should have one, and only one, reason to change.
    In other words, a class should do exactly one job. If a class has multiple responsibilities, those responsibilities become coupled. A change to one responsibility can unexpectedly break or modify the behavior of the other responsibilities.

    A class should be responsible for only one part of the application's functionality, so it should change for only one business reason.

Let us see an example along with Java(Spring boot):
    Imagine you're building a social media application.

You create a service like this:
      @Service
      public class UserService {
        public void registerUser(User user) {
          // validate user
    
          // save user to database

          // send welcome email

          // generate JWT token

          // log registration activity
        }
      }
  
    At first glance, it looks fine. But think about it.
    This class can change because:
        Validation rules changed.
        Database logic changed.
        Email format changed.
        JWT generation changed.
        Logging strategy changed.
So there are 5 different independent reasons for this class to change.

Problem without SRP
    Suppose the company says:
        "Change the welcome email template."
      You modify the email code inside UserService. Now you accidentally break user registration logic.
      Why?
          Because everything is tightly coupled in one class.
      This causes:
          Difficult maintenance
          Difficult testing
          Difficult debugging
          Higher chance of bugs
      This violates SRP.


Applying SRP
    Separate responsibilities. The below is how we should break down the tightly coupled code into achieving the Single Responsibility Principle(SRP).

    Validation.java:
      @Component
      public class UserValidator {
      
          public void validate(User user) {
              // validation logic
          }
      }

    EmailService.java:
      @Service
      public class EmailService {
      
          public void sendWelcomeEmail(User user) {
              // email logic
          }
      }

    JWTService.java
      @Service
      public class JwtService {
      
          public String generateToken(User user) {
              return "jwt-token";
          }
      }

    User Repository
      @Repository
      public interface UserRepository
              extends JpaRepository<User, Long> {
      }

Now for every logic, we have formed different classes and placed codes only necessary for that specific business needs.
So therefore we have successfully achieved the Single Responsibility Principle(SRP).

Now the UserService.java will become:
        @Service
        public class UserService {
        
            private final UserValidator validator;
            private final UserRepository userRepository;
            private final EmailService emailService;
            private final JwtService jwtService;
        
            public UserService(
                    UserValidator validator,
                    UserRepository userRepository,
                    EmailService emailService,
                    JwtService jwtService) {
        
                this.validator = validator;
                this.userRepository = userRepository;
                this.emailService = emailService;
                this.jwtService = jwtService;
            }
        
            public String registerUser(User user) {
        
                validator.validate(user);
        
                userRepository.save(user);
        
                emailService.sendWelcomeEmail(user);
        
                return jwtService.generateToken(user);
            }
        }

After the SRP modification in our code, now each class has their clear responsibility.

Class	                Responsibility
UserValidator	  -      Validation
UserRepository	  -      Database access
EmailService	  -      Email sending
JwtService	      -      JWT handling
UserService	      -      User registration workflow

Advantages:
    Why this is better?
            Why is this better?
                1. Easier Maintenance
                        Suppose JWT implementation changes.
                        Earlier:
                                UserService
                        needed modification.
                        Now:
                            JwtService alone changes.
                        Nothing else is touched.

                2. Easier Testing
                        Without SRP:
                            registerUser()
                        would test:
                            validation, database, email, jwt, logging
                        all together.

                        With SRP:
                            JwtServiceTest tests JWT only.
                            EmailServiceTest tests email only.
                            UserValidatorTest tests validation only.
                        Much simpler.

                3. Reusability
                        Suppose later you add: /login endpoint.
                        JWT generation is already inside:
                                JwtService
                        You simply reuse it.
                            jwtService.generateToken(user);
                        No duplication.


Note:
    Many people say: "One method per class" or "one function per class."
    That is not SRP. A class can have many methods and still follow SRP.
    Example:
        @Service
        public class JwtService {
            public String generateToken(User user) {}
            public boolean validateToken(String token) {}
            public String extractUsername(String token) {}
        }
    This class has 3 methods. Yet it follows SRP because all methods are related to JWT management.
    There is only one reason to change: JWT handling requirements changed.

How to identify SRP violations
Ask yourself:
    "How many independent reasons could force me to modify this class?"
    If the answer is:
        Validation rules, Database changes, Email changes, JWT changes, Logging changes
    then SRP is violated.
    If the answer is:
        JWT requirements changed
    then SRP is likely being followed.

Simple Rule
    In Spring Boot, a good sign of SRP is when classes naturally fit into roles such as:
        Controller → Handle HTTP requests
        Service → Business workflow
        Repository → Database access
        Validator → Validation
        EmailService → Email operations
        JwtService → JWT operations
